# LAVA 集群构建法则

> 本文件是 lava-docker 集群构建/部署的**权威法则**。
> 所有构建、配置、部署操作必须遵守以下条款。
> 法则与代码/其它文档冲突时，**以本文件为准**；需要变更时由用户直接更新本文件。
> 最后更新：2026-06-30（新增法则6：tftp 目录布局）

---

## 法则 1（output 重建）
删除 `output/` 目录后，必须通过 `lavalab-gen.sh` 进行重建，不得手工拼凑 `output/`。

## 法则 2（集群重建的最小拓扑）
集群被完全删除后重建时，重建出的集群**只能有一个 master 和一个 slave**的配置。
设备相关、job 相关的内容一律**不烘进重建集群**，全部通过后续调用 API 添加。

## 法则 3（最小化挂载）
集群镜像尽量少地挂载外部目录，只保留运行时必须持续读取的挂载。

## 法则 4（user-data 为源，脚本随源）
后续需要添加的设备类型、设备字典等内容，放在 `user-data/` 目录下各自的文件夹内；
向集群注入这些内容的脚本，也放在 `user-data/` 文件夹内。

## 法则 5（中间产物用完即清）
注入过程中渲染出的中间产物（如 `user-data/device-dicts/`），在投递进集群、
其内容已落 master 数据库后，必须在使用完成时清理掉，不在工程目录中长期保留。
注入脚本须自带清理逻辑，保证一次注入跑完后工程目录不残留中间产物。

## 法则 6（tftp 目录布局）
`user-data/tftp/` 是被 TFTP/HTTP 服务的运行时数据，整体不入库、可随时删，
仅设备运行时脚本（`*.sh`）与目录骨架（`.gitkeep`）受版本控制。两类用途分目录：
- **下行·易失** `tftp/<device_type>/`：发给设备的产物，用完可清；job 按固定路径下载。
- **上行·缓存** `tftp/_backup/<device_type>/`：job 写回的备份，须 `0777`（tftpput 可写），
  过渡用本地缓存，后续交 Nexus 管理。

job 配套脚本（设备运行时经 HTTP curl 下来执行的，如 flash/backup）的**源**放
`job-templates/<dt>/*.sh`（需按设备/lab 取值时用 `*.sh.j2`），由 `gen-jobs.py` **物化**进
`tftp/<dt>/`；tftp 里那份是产物，删了重跑 `gen-jobs.py` 即恢复。改脚本改源、不改 tftp 副本。

host 侧工具（不发给设备的，如 `inject-ssh-key.sh`）放 `user-data/tools/`，不进服务树。

## 法则 0（法则优先）
